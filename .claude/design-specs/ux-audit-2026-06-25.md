## 🔴 CRITICE — Trebuie rezolvate înainte de orice release

---

### C1 — Token system incomplet: 6+ niveluri de opacitate nestandard pentru text

**Locație:** `styles.css` + fiecare component CSS  
**Problema:** Nu există variabile pentru ierarhia textului. Codul uses direct:
```css
rgba(255,255,255, 0.85)  /* --white-soft — definit */
rgba(255,255,255, 0.55)  /* nedefinit */
rgba(255,255,255, 0.50)  /* nedefinit */
rgba(255,255,255, 0.45)  /* nedefinit */
rgba(255,255,255, 0.40)  /* nedefinit */
rgba(255,255,255, 0.38)  /* nedefinit */
rgba(255,255,255, 0.35)  /* nedefinit */
rgba(255,255,255, 0.32)  /* nedefinit */
rgba(255,255,255, 0.30)  /* nedefinit */
rgba(255,255,255, 0.28)  /* nedefinit */
```
Asta înseamnă că textul "secundar" are opacitate diferită în fiecare component. Imposibil de actualizat consistent.

**Fix:**
```css
/* styles.css — adaugă acești tokeni */
--text-primary:   rgba(255, 255, 255, 1.00);    /* titluri, CTA */
--text-secondary: rgba(255, 255, 255, 0.70);    /* body text */
--text-tertiary:  rgba(255, 255, 255, 0.45);    /* labels, date */
--text-muted:     rgba(255, 255, 255, 0.28);    /* disabled, placeholder */
```
Și înlocuiește toate valorile hardcoded din componente.

---

### C2 — Contrast WCAG sub limita minimă pentru text tertiar și muted

**Locație:** `dashboard.component.css` (gs-date, gs-bmi, gs-longest), `login.component.css` (bs-label, brand-body)  
**Problema:** `rgba(255,255,255, 0.28)` pe `#0d0d10` are contrast ratio ≈ 2.4:1. WCAG AA cere **4.5:1** pentru text normal. Textele afectate sunt citite în contexte importante:
- Data curentă în dashboard greeting (`.gs-date: rgba(255,255,255,.32)` → 2.7:1)
- "Best: Xd" streak (`.gs-longest: rgba(255,255,255,.28)` → 2.4:1)
- Stat labels în auth ("Active Users", "Satisfaction", "App Rating": `rgba(255,255,255,.38)` → 3.2:1)

**Fix:** Ridică opacitatea textului terțiar la minim `0.50` (≈4.5:1 pe `#0d0d10`) și textului muted la minim `0.38` pentru text non-esențial.

---

### C3 — Accent color (#ff4081) are 3 semnificații contradictorii

**Locație:** `styles.css --accent`, `post-card.component.css`, `bottom-nav.component.css`, `feed.component.css`  
**Problema:** Acelați roz/pink este folosit simultan pentru:
1. **Like activ** (inimă liked pe post card)
2. **Badge unread messages** pe bottom nav
3. **Create FAB gradient** (`--nav-create-gradient`)
4. **Stare de eroare/destructive** (`.btn-ghost.danger`, `.post-card-menu-item--danger`)

Roz = acțiune prietenoasă (like, create) ȘI = pericol (delete, eroare). Asta violează principiul Von Restorff — dacă tot e roșu, nimic nu iese în evidență.

**Fix:** Separă semantic colors:
```css
--color-like:        #ff4081;   /* like, reacție pozitivă */
--color-create:      linear-gradient(135deg, var(--primary), #ff4081);
--color-destructive: #ef5350;   /* delete, erori — roșu diferit */
--color-badge:       #ff4081;   /* badge — poate rămâne */
```

---

### C4 — "Forgot password?" este un link mort (`href="#"`)

**Locație:** `login.component.html:119`  
**Problema:**
```html
<a href="#" class="link-muted">Forgot password?</a>
```
Link nefuncțional în producție. Utilizatorii care au uitat parola nu au nicio cale de reset. Implicit, vor încerca parole random, vor eșua și vor abandona app-ul.

**Fix:** Fie implementează ruta `/forgot-password`, fie ascunde link-ul complet până la implementare (`@if (false)`). Nu lăsa UI deceptiv.

---

### C5 — Butoanele Google/Apple sunt decorative, nu funcționale

**Locație:** `login.component.html:141-155`, `register.component.html`  
**Problema:** Butoanele "Continue with Google" și "Continue with Apple" există în UI dar nu au handler OAuth. Utilizatorul le va apăsa și nu se va întâmpla nimic. Asta generează confuzie și neîncredere.

**Fix:** Fie implementează OAuth, fie elimină butoanele. Dacă sunt "coming soon", marchează-le explicit:
```html
<button class="social-btn" disabled title="Disponibil în curând">
  Google <span class="coming-soon-tag">în curând</span>
</button>
```

---

### C6 — `--surface-elevated` nedefinit în token system

**Locație:** `post-card.component.css:308`  
```css
background: var(--surface-elevated, #1a1a24);
```
Variabila nu este definită în `styles.css`. Codul cade întotdeauna pe fallback hardcoded `#1a1a24`. Dacă theme-ul se schimbă, dropdown-ul post-card rămâne cu culoarea veche.

**Fix:** Adaugă în `styles.css`:
```css
--surface-raised:   #111116;  /* cards, panels */
--surface-elevated: #1a1a22;  /* dropdowns, menus */
--surface-overlay:  #20202a;  /* modals */
```

## 🟡 HIGH — Probleme semnificative de UX/DX

---

### H1 — Create action lipsă complet pe desktop

**Locație:** `social-feed.component.html`, `social-feed.component.css`  
**Fluxul rupt:**
- Bottom nav (cu FAB-ul de create) este **vizibil doar pe mobile** (nu are media query pe desktop)  
- `feed-fab` CSS: `display: none` pe ≤768px — ascuns pe mobile (corect, bottom nav îl înlocuiește)
- `feed-header-btn` (butonul "New" din header): `display: none` pe ≤768px — ascuns pe mobile  
- Pe **desktop**: feed-fab e vizibil, dar poate fi în conflict cu layout-ul social shell care are side-nav propriu

Verifică că pe desktop există o cale clară de a crea un post.

---

### H2 — Aspect ratio conflict pe post card images

**Locație:** `post-card.component.html:104`, `post-card.component.css:167-172`  
**Problema:**
```html
<!-- template — inline style: 4/3 -->
<div class="post-card-image-wrap" style="aspect-ratio: 4/3; width: 100%; overflow: hidden">
```
```css
/* CSS component — 3/4 */
.post-card-image-wrap { aspect-ratio: 3/4; }
```
Inline style bate CSS-ul componentului. Imaginile sunt forțate la 4:3 (landscape) indiferent de ce scrie în CSS. Dacă intenția este portrait (3:4 = mai înalt), inline style trebuie eliminat. Dacă intenția este landscape (4:3), CSS-ul componentului trebuie actualizat.

---

### H3 — Orb animations nu respectă `prefers-reduced-motion`

**Locație:** `login.component.css:63-80`, animații `orbFloat1`, `orbFloat2`  
**Problema:** Animațiile de fundal (floating orbs pe brand panel) rulează permanent fără nicio condiție `prefers-reduced-motion`. Utilizatorii cu epilepsie photosensibilă sau vestibular disorders vor vedea mișcare continuă în câmpul lor vizual.

**Fix:**
```css
@media (prefers-reduced-motion: reduce) {
  .orb { animation: none; }
  .brand-logo { animation: none; }
}
```

---

### H4 — Profile bio clamped la 2 rânduri fără "Show more" pe profilul propriu

**Locație:** `social-profile.component.css:103-107`  
```css
.profile-bio {
  -webkit-line-clamp: 2;
  overflow: hidden;
}
```
Pe profilul propriu, userul vede bio-ul trunchiat la 2 rânduri. Nu există niciun buton "Expand" sau "Read more" pe view mode. Userul nu poate citi ce a scris întreg.

---

### H5 — Inline styles în template-uri

**Locație:** `social-profile.component.html:295`  
```html
<span style="font-size: 11px; opacity: 0.5;">{{ workout.type }} · {{ workout.durationMin }} min</span>
```
Style-urile inline nu pot fi themate, overriden sau actualizate consistent. Mută în clasa CSS a componentului.

---

### H6 — Legacy Angular syntax (`*ngIf`, `*ngFor`) mixat cu syntax nou (`@if`, `@for`)

**Locație:** `hero-slider.component.html` (tot fișierul)  
Hero slider folosește `*ngFor`, `*ngIf`, `*ngIf="!..." else` — sintaxa pre-Angular 17. Restul aplicației folosește `@if`, `@for`. Pe lângă inconsistența vizuală în cod, Angular 19 va depreca sintaxa veche.

---

### H7 — Feed skeleton nu corespunde cu forma posturilor reale

**Locație:** `social-feed.component.css:97-112`  
```css
.feed-skeleton-card {
  height: 200px; /* box simplu */
  border-radius: 16px;
}
```
Posturile reale sunt flat (fără border-radius, fără card elevation), cu avatar + header + content + footer. Skeleton-ul este un dreptunghi rotunjit generic. La încărcare, există un layout shift vizibil și confuzant.

---

### H8 — `focus-visible` lipsă pe bottom nav tabs

**Locație:** `app-bottom-nav.component.css`  
Fiecare `.app-bottomnav-tab` nu are `:focus-visible` styling. Utilizatorii care navighează cu tastatura nu pot vedea unde se află focus-ul pe navigare — element critic pentru accesibilitate.

**Fix:**
```css
.app-bottomnav-tab:focus-visible {
  outline: 2px solid var(--primary);
  outline-offset: -2px;
  border-radius: 8px;
}
```

---

### H9 — `z-index: 100` pe dropdown-ul post-card e sub nav (z-index: 1000)

**Locație:** `post-card.component.css:312`  
Dacă un dropdown de post options se deschide lângă nav sau lângă un modal (z-index: 900+), meniu-ul dispare în spatele lor. `z-index: 100` este prea jos.

**Fix:** Ridică la `z-index: 200` sau mai bine folosește `position: fixed` pentru dropdown-uri care pot ieși din container.

## 🟡 MEDIU — Probleme de craft și consistență

---

### M1 — Border-radius complet inconsistent fără sistem

Valorile găsite în codebase (nu exhaustiv):
- Butoane: `8px`, `10px`, `12px`, `14px`
- Cards: `12px`, `14px`, `16px`, `20px`, `24px`
- Avatare: `50%`
- Icoane wrapper: `10px`, `12px`, `14px`, `16px`
- Inputs: `10px` (MDC token), `12px` (modal-box), `20px` (modal)

Nu există nicio documentație sau token standard. "Soft" sau "sharp"? Nu se știe.

**Fix propus:**
```css
--radius-sm:   8px;   /* butoane inline, badge */
--radius-md:   12px;  /* cards, inputs, dropdowns */
--radius-lg:   16px;  /* modal-uri, sheet-uri */
--radius-xl:   24px;  /* bottom sheets, large panels */
--radius-full: 9999px; /* pills, avatare */
```

---

### M2 — Scrollbar custom la 14px lățime — prea mare

**Locație:** `styles.css:155-183`  
Scrollbar-ul webkit este setat la `width: 14px`. Standard este 6–8px. La 14px mănâncă din spațiul de layout pe desktop și arată nepolit pe ecrane mari.

**Fix:** 
```css
::-webkit-scrollbar { width: 6px; height: 6px; }
```

---

### M3 — Gradient text overuse diluează impactul

Elementele cu gradient text (`-webkit-background-clip: text`):
1. Auth headline "right here." (justificat — hero accent)
2. Auth stats values "12k+", "98%", "4.9" (acceptabil)
3. Home page stat values (repetă aceeași mecanică)
4. Dashboard greeting `gs-name` — **nu e necesar**
5. `form-title` "Welcome back" — gradient text pe un titlu H1 de formular

Când totul e gradient, nimic nu iese în evidență. Rezervă pentru maximum 1–2 elemente per ecran.

---

### M4 — `.navbar { height: 12vh }` — clasă orfană în styles.css

**Locație:** `styles.css:105-107`  
```css
.navbar {
  height: 12vh;
}
```
Această clasă nu mai este folosită în niciun template (navigarea folosește `--nav-height`). Este un artefact de la un design anterior. Trebuie ștearsă.

---

### M5 — Avatar sizes nestandard

| Context | Dimensiune |
|---|---|
| Post card (feed) | 36×36px cu border 2px |
| Profile page | 72×72px cu border 3px |
| Social side nav | Probabil altă dimensiune |
| Blog post card | Există `blog-post-card-avatar` |

Nu există nicio variabilă sau documentație pentru dimensiunile avatarelor. Un sistem minimal:
```css
--avatar-sm:  32px;  /* inline în text */
--avatar-md:  40px;  /* feed posts, lists */
--avatar-lg:  72px;  /* profile header */
--avatar-xl:  96px;  /* profile hero */
```

---

### M6 — Dashboard page fără grid layout explicit

**Locație:** `dashboard-page.component.html`  
```html
<div class="page-wrapper">
  <app-header></app-header>
  <div class="page-content">
    <app-dashboard></app-dashboard>
    <app-daily-user-data></app-daily-user-data>
    <app-previous-daily-user-data></app-previous-daily-user-data>
  </div>
  <app-footer></app-footer>
</div>
```
Fiecare component gestionează singur `max-width` și `padding` (dashboard: `max-width: 1100px`, daily-user-data: probabil altul). Rezultatul: inconsistență de aliniere între secțiuni pe desktop.

**Fix:** Container comun în `page-content` cu `max-width` și `padding` centralizate.

---

### M7 — Profile avatar prea mic (72px) față de standardele industriei

Pe profilul social, avatarul principal este 72×72px. Instagram: 110px. Twitter/X: 96px. La 72px pe un display Retina este ok, dar pe ecrane mobile standard arată mic și greu de atins (Fitts's Law: target-ul e sub 44pt recomandat).

Recomandare: Ridică la 88px sau 96px.

## 🟢 CE FUNCȚIONEAZĂ BINE

Aceste zone sunt implementate corect și trebuie păstrate ca referință:

### ✅ Z-index map documentat în styles.css
Comentariile cu stack-ul z-index (920, 930, 1000, 1050, etc.) sunt excelente. Rareori se vede asta documentat explicit.

### ✅ Safe area insets implementate corect
`env(safe-area-inset-top)` și `env(safe-area-inset-bottom)` sunt aplicate pe top bar, bottom nav și page-wrapper. Funcționează corect pe iPhone cu notch.

### ✅ `prefers-reduced-motion` în bottom nav
```css
@media (prefers-reduced-motion: reduce) {
  .bottomnav-create { transition: none; }
}
```
Bine implementat. Trebuie extins la restul animațiilor.

### ✅ ARIA labels pe navigare
Bottom nav, top bar, post card actions au `aria-label` și `aria-pressed`/`aria-current` corecte.

### ✅ Skeleton screens și guided empty states
Existența skeleton-urilor și a stărilor de empty state structurate (guided-empty) este un semn de maturitate. Problema este că skeleton-ul nu reflectă fidel forma content-ului (H7 de mai sus).

### ✅ Login split-screen glassmorphism
Design-ul autentificării este cel mai lustruit din app. Hierarhie clară, animații fine, brand panel vs form panel bine separate.

### ✅ Post card — states complete
Like, comment, follow, edit, delete, archive — toate acțiunile sunt prezente. `aria-pressed` pe like e corect.

### ✅ `cubic-bezier(0.34, 1.56, 0.64, 1)` pe create FAB
Spring easing este alegerea corectă pentru un buton de acțiune primară. Micro-interacțiunea e plăcută.

## Token system recomandat (adaugă în styles.css)

```css
:root {

  --text-secondary:  rgba(255, 255, 255, 0.70);
  --text-tertiary:   rgba(255, 255, 255, 0.50);
  --text-muted:      rgba(255, 255, 255, 0.32);
  --text-disabled:   rgba(255, 255, 255, 0.22);

  /* ── Surface elevation ── */
  --surface-base:     #0d0d10;
  --surface-raised:   #111116;
  --surface-elevated: #1a1a22;
  --surface-overlay:  #20202a;

  /* ── Border ── */
  --border-subtle:  rgba(255, 255, 255, 0.04);
  --border-default: rgba(255, 255, 255, 0.08);
  --border-strong:  rgba(255, 255, 255, 0.14);

  --color-destructive: #ef5350;

  /* ── Radius scale ── */
  --radius-sm:   8px;
  --radius-md:   12px;
  --radius-lg:   16px;
  --radius-xl:   24px;
  --radius-full: 9999px;

  /* ── Avatar sizes ── */
  --avatar-sm: 32px;
  --avatar-md: 40px;
  --avatar-lg: 88px;
  --avatar-xl: 108px;
}
```
